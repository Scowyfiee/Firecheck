## 🛠️ 第一章：实验目的与背景

### 1. 实验目的
*   深入理解并掌握“端-边-云”三层架构在分布式实时智能监控系统中的工程应用。
*   掌握基于 YOLOv11 目标检测算法的火焰与烟雾多尺度目标实时检测及置信度阈值调优方法。
*   实现轻量化、极低延迟的实时视频预览通道（基于 WebSocket 的 Raw Binary/Blob 帧流分发）。
*   解决边缘计算端 OpenCV 视频捕获与 HTML5 浏览器播放器不兼容的底层问题，实现基于 FFmpeg 异步多线程转码（libx264/libopenh264 自动降级）与 Flask 服务端 HTTP 206 Range 范围请求的视频回放管道。
*   设计网络代理（如 Clash 代理）环境下的自适应绕过路由（Bypass），保障复杂演示网络环境下的通信可用性。

### 2. 实验背景
火灾检测作为公共安全与工业监控领域的重中之重，传统的“烟感”、“温感”传感器受限于物理介质扩散速度，在野外、森林、大空间厂房等开阔区域经常出现报警滞后甚至失效的问题。

近年来，随着计算机视觉与深度学习的快速演进，利用 YOLO 系列模型进行火焰和烟雾的联合实时检测已成为主流。同时，为避免将所有的海量实时视频流送至云端处理导致的巨大网络带宽开销，设计将 AI 推理下沉至边缘设备（如 ARM 架构 Orange Pi 5 等），就地计算并仅将结构化报警元数据（报警图、转码短录像、坐标、置信度等）上报至云端进行二级审核的“端边云协同”架构，具有极高的学术研究与商业应用价值。

### 1. YOLOv11 火焰烟雾联合检测算法
系统加载经 3000 张包含真实场景下火焰和阴燃烟雾图像训练而成的 YOLOv11 目标检测模型。
*   **输入尺寸优化**：在 `detect_frame` 阶段指定参数 `imgsz=480`，相较于默认的 640，此项改动在轻微牺牲边缘极小物体精度的代价下，使边缘端推理耗时下降了约 40%，在无显卡硬件的边缘 CPU 下依然可保持约 24 FPS 的流畅检测。
*   **多尺度置信度调优**：火焰和烟雾分类权重采用并联逻辑，当任意类别置信度（Confidence Score）大于或等于 0.35 时，即判定为疑似火情，自动导入后端的过滤判定逻辑。

### 2. 轻量化 WebSocket 原生二进制图像推送设计
为消除流媒体协议高延迟对火警处置的致命危害，系统自主设计了基于二进制推送的实时画面传递：
*   边缘端在子线程中维护一个独立的 WebSocket 服务（默认端口 9999 / 可通过交互指定）。
*   在主检测循环中，YOLO 标注好的视频帧通过 `cv2.imencode('.jpg', frame, [cv2.IMWRITE_JPEG_QUALITY, 50])` 压缩为高压缩率的二进制 JPEG 数据。
*   直接通过 WebSocket 的 `binary` 帧分发至所有连接的前端客户端。前端大屏获取 Blob 数据后利用 `window.URL.createObjectURL(blob)` 转换为图片引用，消除一切文本编解码过程，耗时维持在毫秒级。

### 3. 多事件并发线程安全录像与 FFmpeg 转码降级管道
OpenCV 默认导出的 `mp4v`（MPEG-4）视频文件无法在主流 Web 浏览器（Chrome/Safari/Edge）的原生 `<video>` 标签中解码播放。本系统对此进行了底层级的工程攻关：
*   **线程安全并发（多活跃写入器）**：为防止密集并发火警场景下，单个全局 VideoWriter 造成的写锁死、写冲突问题，创建了 `active_writers` 列表和 `_frame_lock` 线程锁。每次发生报警时动态生成专属的录像参数字典，将其追加到 `active_writers` 中，互不干扰地写入本地视频帧。
*   **FFmpeg 编码器自动降级转码管道**：短视频捕获完毕后，系统自动抛出异步任务执行系统级 FFmpeg 进程转码。为了防止某些嵌入式平台缺乏商业授权的 `libx264` 编码器而导致转码失败，设计了**降级回退机制**。如果 `libx264` 调用返回非零，系统自动捕获异常并降级回退至 `libopenh264` 再次尝试，转码后的输出视频格式统一为 `yuv420p` 像素格式的 H.264 Baseline Profile，彻底攻克了“视频无法播放/黑屏”的顽疾。

### 4. 基于局部环回 bypass 的网络代理抗干扰设计
在运行 Clash 等系统代理的本地机器上，回环地址（`127.0.0.1`）以及内网 IP 段的流量经常被代理强行劫持引发网络超时或中断。本系统在设备端向服务端发送 HTTP 心跳与报警信息的请求底层，显式绑定了 `proxies={"http": None, "https": None}` 参数，从系统库底层彻底屏蔽代理的干涉，显著提高了系统的环境自适应能力。

### 1. 线程安全并发录制与转码设计 (`flame_detect.py`)
以下是我们在边缘设备端实现的核心 `start_recording` 和 `stop_recording` 转码管道代码片段：

```python
    def start_recording(self, frame_w=1280, frame_h=720):
        # 1. 动态生成文件名与存储路径
        ts = datetime.now().strftime("%Y%m%d_%H%M%S")
        filename = f"alarm_{ts}.mp4"
        filepath = os.path.join(self.cfg.save_dir, filename)
        fps = getattr(self, "fps", 20.0)
        if fps <= 0: fps = 20.0

        # 2. 优先使用标准 AVC1/H264 四字符码，若失败则采用 MP4V 兼容写入
        try:
            fourcc = cv2.VideoWriter_fourcc(*"avc1")
            writer = cv2.VideoWriter(filepath, fourcc, fps, (frame_w, frame_h))
            if not writer.isOpened(): raise Exception("avc1 writer not opened")
        except Exception:
            fourcc = cv2.VideoWriter_fourcc(*"mp4v")
            writer = cv2.VideoWriter(filepath, fourcc, fps, (frame_w, frame_h))

        # 3. 回写 60 帧历史缓冲区的视频，提供上下文
        for f in list(self.frame_buffer):
            fh, fw = f.shape[:2]
            if fw == frame_w and fh == frame_h:
                try: writer.write(f)
                except Exception as e: logger.error(f"Error writing buffered frame: {e}")
        self.frame_buffer.clear()

        # 4. 将该次独立报警的写入器信息组装成字典放入活跃列表
        writer_entry = {
            "writer": writer, "w": frame_w, "h": frame_h,
            "filepath": filepath, "filename": filename, "start_time": time.time()
        }
        with self._frame_lock:
            self.active_writers.append(writer_entry)
            self.recording = True
        return writer_entry
```

```python
    def stop_recording(self, writer_entry=None):
        targets = [writer_entry] if writer_entry is not None else list(self.active_writers)
        last_filepath, last_filename = "", ""

        for entry in targets:
            writer = entry["writer"]
            filepath = entry["filepath"]
            filename = entry["filename"]

            # 释放当前写入器
            try: writer.release()
            except Exception as e: logger.error(f"Error releasing writer: {e}")

            # 移出活跃列表
            with self._frame_lock:
                if entry in self.active_writers: self.active_writers.remove(entry)
                if not self.active_writers: self.recording = False

            last_filepath, last_filename = filepath, filename

            # FFmpeg 编码器自动降级转码逻辑
            if filepath and os.path.exists(filepath) and os.path.getsize(filepath) > 0:
                temp_path = filepath + ".tmp.mp4"
                ffmpeg_cmd = "ffmpeg"
                # 优先搜寻系统路径
                for p in ["/usr/bin/ffmpeg", "/usr/local/bin/ffmpeg"]:
                    if os.path.exists(p) and os.access(p, os.X_OK):
                        ffmpeg_cmd = p
                        break

                transcode_success = False
                for encoder in ["libx264", "libopenh264"]:
                    try:
                        cmd = [
                            ffmpeg_cmd, "-y", "-i", filepath,
                            "-vcodec", encoder, "-pix_fmt", "yuv420p",
                            "-profile:v", "baseline", "-level", "3.0", temp_path
                        ]
                        res = subprocess.run(cmd, stdout=subprocess.PIPE, stderr=subprocess.PIPE, timeout=35)
                        if res.returncode == 0 and os.path.exists(temp_path) and os.path.getsize(temp_path) > 0:
                            os.replace(temp_path, filepath)
                            logger.info(f"Video transcoded successfully with {encoder}")
                            transcode_success = True
                            break
                    except Exception as e:
                        logger.warning(f"Encoder {encoder} failed: {e}")
                    if os.path.exists(temp_path):
                        try: os.remove(temp_path)
                        except: pass
        return last_filepath, last_filename
```

### 2. 支持 Range 分段加载的 Flask 媒体读取服务 (`web_server.py`)
利用 Flask 手动拦截 Range 头部，返回 206 Partial Content 以实现视频的秒开与无阻碍进度条拖拽：

```python
@app.route("/uploads/<path:filename>")
def uploaded_file(filename):
    filepath = UPLOAD_DIR / filename
    if not filepath.exists(): return abort(404)

    # 仅针对视频资源应用分块读取
    if not filename.lower().endswith((".mp4", ".mov", ".avi", ".mkv")):
        return send_from_directory(str(UPLOAD_DIR), filename)

    file_size = filepath.stat().st_size
    range_header = request.headers.get("Range", None)
    if not range_header:
        return send_from_directory(str(UPLOAD_DIR), filename)

    # 解析请求字节范围 (如 Range: bytes=1024-2048)
    byte1, byte2 = 0, None
    m = re.search(r"bytes=(\d+)-(\d*)", range_header)
    if m:
        g = m.groups()
        if g[0]: byte1 = int(g[0])
        if g[1]: byte2 = int(g[1])

    length = file_size - byte1
    if byte2 is not None and byte2 != "":
        try: length = int(byte2) - byte1 + 1
        except ValueError: pass

    # 读取特定区间的视频字节
    with open(filepath, "rb") as f:
        f.seek(byte1)
        data = f.read(length)

    rv = Response(data, 206, mimetype="video/mp4", direct_passthrough=True)
    rv.headers.add("Content-Range", f"bytes {byte1}-{byte1 + len(data) - 1}/{file_size}")
    rv.headers.add("Accept-Ranges", "bytes")
    return rv
```

## 🏁 第六章：实验结论与展望

### 1. 实验结论
本实验成功搭建并跑通了基于 YOLOv11 目标检测算法的智能消防“端-边-云”协同系统。
*   通过基于 WebSocket 的局域网异步扫描握手，实现了边缘监控设备零配置、零争抢的即插即用和自动上线注册。
*   通过研究并实现线程安全的并发录像和 FFmpeg 降级编码转码机制，彻底消除了网页端点播 OpenCV 默认录像黑屏的现象。
*   利用 Flask 的 HTTP 206 Partial Content 支持，大幅降低了弱网下报警详情回放视频的首屏加载耗时，成功实现了视频进度的自由定位。
*   设备端显式配置代理 bypass 规则，有效规避了演示环境下的 Clash 全局代理网络阻塞报错。

### 2. 不足与改进展望
*   **嵌入式加速不够充分**：目前 YOLOv11 仍运行于 Orange Pi 5 的 CPU 线程，未能完全激活其内置的 4Tops NPU 硬件加速器。未来工作重点是将 PyTorch 权重转化为 RKNN 格式，全量运行于 NPU 推理器上，以期待推理速度飙升至 100 FPS 以上。
*   **跨网段自发现限制**：目前的前端端口扫描自发现机制仅局限于同一局域网网段下。对于跨网段或广域网部署的监控节点，下一步建议开发基于 MQTT 中继的云端路由映射，从而拓宽系统的整体可用范围。
